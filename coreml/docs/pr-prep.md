## 1. What this PR adds

A new tract crate, **`tract-coreml`**, that lets tract dispatch supported subgraphs of a `TypedModel` to **Apple Core ML** on macOS / iOS — closing tract's biggest remaining hardware gap on Apple Silicon (no path to the Neural Engine until now).

Sibling to `tract-metal` and `tract-cuda`. Registered through the standard `tract_core::Runtime` trait. Opt-in via workspace member `coreml`; not in `default-members`.

### Key architectural pieces

- **Pure-Rust runtime MIL build** — no Python `coremltools` dependency, no system `protoc`. Schema vendored from `apple/coremltools` 9.0; codegen via `prost-build` + `protox`.
- **Subgraph fusion** — translatable nodes get partitioned into maximal connected subgraphs (union-find with all-or-nothing + convex-region rules); each subgraph becomes one MLPackage / one `CoremlOp` node. Designed for ANE engagement (Apple's scheduler decides per-MLPackage).
- **Persistent compile cache** — `~/Library/Caches/tract-coreml/v1/<sha256>.mlmodelc/`. Canonical proto walking handles `prost::encode` HashMap iteration non-determinism.
- **In-MLPackage rank-4 padding convention** — every translator pads to rank ≥ 4 (with leading 1s) so chains stay rank-consistent inside a single MLPackage; defensive input-reshape pattern in every emit.
- **External-shape boundary mechanism** — translators expose `*_external_shape` fields that strip leading unit dims at the MLPackage I/O boundary, letting tensors with rank > 5 (Hiera windowed-attention 6D forms with declutter-inserted decorator axes) flow through MIL's rank-5 cap. CoremlOp does the squeeze/unsqueeze at the boundary.
- **Per-input MIL dtype** — F16 default for floats, I32 accepted for indices, **I64 → I32 narrowing cast at the CoremlOp eval boundary** (MLMultiArray doesn't natively support Int64). Required for any LLM (token IDs are I64).
- **Compile-cache key** is the SHA-256 of `(MIL bytes, weight blob bytes)` after a canonical proto walk that sorts HashMap keys — handles `prost`'s non-deterministic map encoding.
- **No `tract-gpu` dependency** — Core ML's MLMultiArray ownership and per-prediction memory model don't fit the gpu-shared abstractions; tract-coreml is self-contained.
- **MLProgram only** (no NeuralNetwork format). **MIT-OR-Apache-2.0** to match the rest of the workspace.

## 4. Code composition (defensive-code attribution)

`tract-coreml` is **18,171 LoC total** (`src/` 11,410 + `tests/` 6,730 + `build.rs` 31). Of the source, **roughly 37% is defensive code working around tract / MIL / Apple friction** rather than intrinsic CoreML translator complexity. This is a strong signal for the upstream proposals in §6: every accelerator backend (tract-metal, tract-cuda, this PR's tract-coreml) carries the same defensive workarounds, so upstream changes amortize across all of them.

### 4.3 Potential reduction if upstream issues land

Three highest-leverage upstream wins, ranked by LoC payoff:

| Upstream change | tract-coreml shrinks by | What collapses |
|---|---:|---|
| **I5 + I9** (canonical EinSum form; stop emitting decorated signatures) | **~1,500 LoC** | `general_matmul` + `matmul` + `einsum` + `einsum_outer` (1,832 LoC today) collapse to a single ~300 LoC canonical-form einsum translator |
| **I4 + I7 + I13** (preserve high-level norm ops; stop decomposing InstanceNorm / LayerNorm / Norm chains) | **~800 LoC** | Pattern detectors (`instance_norm.rs` + `layer_norm.rs`) disappear; norm translators become ~50 LoC each instead of 290+ |
| **A2** (fold `Cast(Const)` chains after `f32_to_f16`) | **~400 LoC** | The `*Input::Const` enums + per-translator Const handling simplify dramatically |
| **Combined potential reduction** | **~2,700 LoC (~24% of current `src/`)** | tract-coreml would land at ~8,700 LoC of source |

**Why this matters for the PR conversation**: every accelerator backend (tract-metal, tract-cuda, the proposed tract-coreml) carries the same ~2,700 LoC of duplicated defensive workarounds. **One upstream change collapses that work across N backends.** This is the central architectural argument behind the I4 / I5 / I7 family of upstream proposals in §6.

A note on what's NOT defensive: the `mil/` subdir (~390 LoC) is genuine MIL proto wrangling that any backend would need; the `coreml_op.rs` runtime + `context.rs` + `tensor.rs` (~730 LoC) are CoreML-runtime integration that's intrinsic; per-op MIL emit logic (`emit_*_mil` functions) is genuine translation work. The defensive 37% is the price tag of bridging two IRs that disagree on conventions; not a code-quality issue with the backend itself.

**Why 7 of 12 canaries now have a tract-CPU baseline (was 2):** the local Resize-F16 patch landed during this measurement pass. It's a ~10-line change in `tract/onnx-opl/src/resize.rs` plus 2 unit tests. See I1 in upstream-feedback; ready as a standalone upstream PR.

**The DFN3 1.06× speedup is the most informative row in this table.** DFN3 is dominated by 2× `tract_core_scan` loops containing GRU cells — those loops stay on CPU because MIL has no `scan` op (only `while_loop`, which requires either a pattern detector or upstream RNN preservation to take advantage of — see "How to make DFN3/Parakeet/Nemotron run on ANE" in the PR conversation). CoreML's contribution to DFN3 is only the *outside-scan* compute (input Linear projections + decoder convs + output projections), which is ~5% of the wall time. The rest is the recurrent loop running on tract-CPU. **The 1.06× ratio is exactly the empirical answer to "what does our current CoreML backend buy for recurrent audio models?" — it's effectively nothing until either Path B (pattern detector → MIL `while_loop`) or Path C (upstream RNN preservation handout) lands.** This is the single best data point in the corpus for justifying the RNN-preservation handout: 5-15× of potential speedup is currently latent.

**Methodology note on the new CPU-vs-CoreML rows.** The four newly-baselineable canaries (MODNet/RVM/InceptionV3/Real-ESRGAN) were measured by `*_perf_cpu_vs_coreml` tests that run the CPU iterations *before* the CoreML iterations on the same model load. Because the CPU baseline takes seconds-to-minutes (e.g. 4 s on InceptionV3, 12 s on MODNet, 5 s on Real-ESRGAN), the system is thermally hot when the CoreML measurement begins, which can inflate CoreML latency vs a cold-system measurement. Concretely InceptionV3 measures 12.18 ms in this test vs 5.54 ms in the matrix's standalone `inception_v3_perf_coreml_only` test (cold-cache, no preceding CPU run). The **direction of the speedup is unaffected** but the magnitude is conservative: the true CPU-vs-CoreML speedup at steady-state cold-system conditions is likely 2× higher than reported here. MobileNet / SqueezeNet aren't affected (no Resize → CPU baseline was already measureable, and existed in a dedicated test).

### 5.1 Open follow-ups surfaced by this measurement run

1. **DFN3 fragmentation regression (3 → 6 CoremlOps).** Net perf still well under real-time (RTF 0.07 = 14× faster than real-time at 1 s chunks), but the fragmentation around the 2 GRU `tract_core_scan` boundaries grew. The split is happening on the gate sibling chains (Wz / Wh / Wr), each becoming its own subgraph because the convex-region check forbids merging across the Scan boundaries that sit between them. Worth a `git log` walk on `fusion.rs` since the 3-CoremlOp baseline to either re-merge or document. Even un-fixed, this isn't a release blocker — it's a degradation on a metric we already weren't optimising. (Background context: same architectural class as the [RNN-preservation handout](./tract-rnn-preservation-handout.md) addresses end-to-end.)

2. **`MILCompilerForANE FAILED` — same symptom on SAM 2 + SmolLM2 (different models).** Stderr shows `E5RT encountered an STL exception. msg = MILCompilerForANE error: failed to compile ANE model using ANEF. Error=_ANECompiler : ANECCompile() FAILED.` 3 occurrences during SAM 2 compile, 1 during SmolLM2 compile. The two models share nothing structurally except "the MIL graph contains shapes / op patterns Apple's ANE compiler refuses." For SAM 2 the suspect is rank-5/6 reshape patterns (the windowed-attention residuals). For SmolLM2 the suspect is `scatter_nd` + `equal` (bool) + `select` (bool cond) — the auxiliary mask/position-id ops. **Single root-cause investigation** would be cheap: capture both `mlmodelc/coremldata.bin` failures with `MLCustomConfig` ANE-only mode and `xcrun coremlcompiler` -v to get a per-op breakdown of which MIL op triggers the rejection. Could feed an upstream feature request to Apple or — more practically — a model-side rewrite hint for downstream LLM users.

3. **Inception V3 has 1 stranded `EinSum`.** Likely the multi-branch concat split point that creates the 2-CoremlOp shape. Adding it should give a single mega-CoremlOp.

4. **SmolLM2 has 2 stranded `Cast(I64)` ops.** Both around RoPE table indexing. Cast translator currently rejects I64 input dtype; lifting that restriction (with the same I64→I32 boundary cast we use on Gather) would close the last gap and give a truly whole-graph mega-CoremlOp.

5. **Parakeet/Nemotron joint stranded EinSums (3 each).** Final-head shape; probable axes-signature gap in `general_matmul`'s pattern matcher.

6. **Steady-state perf for ViT / YOLOv8 / RealESRGAN / SmolLM2 not yet measured.** Probes captured first-run latency only because the existing coverage tests don't call `state.run` in a loop. A small follow-up to add `*_perf_*` test functions would let those rows in the headline table get filled in. (No ANE engagement check yet either.)

#### Why this framing matters for the PR

Three rhetorical asymmetries the maintainer should understand:

1. **The defensive-LoC argument from §4 is multi-backend.** The ~2,700 LoC of duplicated workarounds isn't unique to tract-coreml — every accelerator backend re-implements the same pattern detectors, the same EinSum canonicalisation, the same Const absorption. Sonos owns three backends already; future backends (XPU, NPU, OpenVINO, Vulkan, etc.) will pay the same tax.
2. **Several items are CPU-side correctness or completeness bugs**, not just backend-author conveniences:
   - **B3** (F16 RmsNorm CPU eval errors) — anyone testing F16 inference hits this.
   - **I1** (CPU Resize F32-only) — anyone running F16 segmentation hits this.
   - **I13** (LayerNorm → RmsNorm rewrite) — *if* this is a correctness bug, every transformer in tract is affected.
   These should be addressed regardless of whether any new backend ever lands.
3. **B8 is a strategic gate**, not a backend convenience. Without `com.microsoft` domain support, **tract is locked out of the modern Microsoft LLM corpus entirely** — Phi-2/3/3.5/4, the de facto standard ONNX format for production LLM inference. This is the highest-leverage single change for tract's competitive positioning vs onnxruntime / ggml / candle.

Where I'm uncertain about the wider impact, I've flagged with ⚠️. Particularly:
- Whether `tract-linalg` would actually grow vendor fastpaths to capitalize on I4-class preservation (depends on roadmap I don't see).
- Whether the NNEF audience (B5 / B6) is large enough outside Sonos's own customer base to motivate work prioritization.
- Whether I13 is genuinely a correctness bug or a clever optimization with a specific valid pattern that the agent's probe didn't see.

These are open questions for the maintainer to weigh. The PR conversation is the right venue.
