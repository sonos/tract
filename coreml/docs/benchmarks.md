## 5. Performance: tract CPU vs tract CoreML

> **Resize-F16 unblock (2026-05-12):** tract's CPU `Resize::eval` was F32-only (upstream issue I1) which previously blocked tract-CPU baselines for MODNet / RVM / InceptionV3 / Real-ESRGAN through the FP16 pipeline. Patched locally in `tract/onnx-opl/src/resize.rs` (cast to F32 internally, cast back to original dtype at the boundary; +F16 unit tests). All four models now have a tract-CPU baseline. See I1 in upstream-feedback.

- MobileNet v2: 3.4 ms → 2.22 ms (35% faster), 128× → 191× speedup, 2 → 1 CoremlOp
- SqueezeNet: 0.7 ms → 0.46 ms (35% faster), 33× → 50× speedup

- RVM: 5.9 ms → 4.86 ms (18% faster), still 1 CoremlOp; **plus newly-baselineable: 111× over tract CPU**
- InceptionV3: now baselineable too — **34× over tract CPU** at default `All`; per-mode A/B confirms the ANE path delivers the bulk of that
- Real-ESRGAN x4: now baselineable — **12.7× over tract CPU**; the lower ratio reflects the upsample-chain CPU residuals (I14)
- SAM 2: still 1.6–2.0 s, still 39 CoremlOps, ANE compile still fails

**Not measured this run** (require new `*_perf_*` test functions): steady-state mean / RTF / sustained loops for ViT, YOLOv8, SmolLM2.

**Why 7 of 12 canaries now have a tract-CPU baseline (was 2):** the local Resize-F16 patch landed during this measurement pass. It's a ~10-line change in `tract/onnx-opl/src/resize.rs` plus 2 unit tests. See I1 in upstream-feedback; ready as a standalone upstream PR.

**The DFN3 1.06× speedup is the most informative row in this table.** DFN3 is dominated by 2× `tract_core_scan` loops containing GRU cells — those loops stay on CPU because MIL has no `scan` op (only `while_loop`, which requires either a pattern detector or upstream RNN preservation to take advantage of — see "How to make DFN3/Parakeet/Nemotron run on ANE" in the PR conversation). CoreML's contribution to DFN3 is only the *outside-scan* compute (input Linear projections + decoder convs + output projections), which is ~5% of the wall time. The rest is the recurrent loop running on tract-CPU. **The 1.06× ratio is exactly the empirical answer to "what does our current CoreML backend buy for recurrent audio models?" — it's effectively nothing until either Path B (pattern detector → MIL `while_loop`) or Path C (upstream RNN preservation handout) lands.** This is the single best data point in the corpus for justifying the RNN-preservation handout: 5-15× of potential speedup is currently latent.

**Methodology note on the new CPU-vs-CoreML rows.** The four newly-baselineable canaries (MODNet/RVM/InceptionV3/Real-ESRGAN) were measured by `*_perf_cpu_vs_coreml` tests that run the CPU iterations *before* the CoreML iterations on the same model load. Because the CPU baseline takes seconds-to-minutes (e.g. 4 s on InceptionV3, 12 s on MODNet, 5 s on Real-ESRGAN), the system is thermally hot when the CoreML measurement begins, which can inflate CoreML latency vs a cold-system measurement. Concretely InceptionV3 measures 12.18 ms in this test vs 5.54 ms in the matrix's standalone `inception_v3_perf_coreml_only` test (cold-cache, no preceding CPU run). The **direction of the speedup is unaffected** but the magnitude is conservative: the true CPU-vs-CoreML speedup at steady-state cold-system conditions is likely 2× higher than reported here. MobileNet / SqueezeNet aren't affected (no Resize → CPU baseline was already measureable, and existed in a dedicated test).

### 5.1 Open follow-ups surfaced by this measurement run

1. **DFN3 fragmentation regression (3 → 6 CoremlOps).** Net perf still well under real-time (RTF 0.07 = 14× faster than real-time at 1 s chunks), but the fragmentation around the 2 GRU `tract_core_scan` boundaries grew. The split is happening on the gate sibling chains (Wz / Wh / Wr), each becoming its own subgraph because the convex-region check forbids merging across the Scan boundaries that sit between them. Worth a `git log` walk on `fusion.rs` since the 3-CoremlOp baseline to either re-merge or document. Even un-fixed, this isn't a release blocker — it's a degradation on a metric we already weren't optimising. (Background context: same architectural class as the [RNN-preservation handout](./tract-rnn-preservation-handout.md) addresses end-to-end.)

2. **`MILCompilerForANE FAILED` — same symptom on SAM 2 + SmolLM2 (different models).** Stderr shows `E5RT encountered an STL exception. msg = MILCompilerForANE error: failed to compile ANE model using ANEF. Error=_ANECompiler : ANECCompile() FAILED.` 3 occurrences during SAM 2 compile, 1 during SmolLM2 compile. The two models share nothing structurally except "the MIL graph contains shapes / op patterns Apple's ANE compiler refuses." For SAM 2 the suspect is rank-5/6 reshape patterns (the windowed-attention residuals). For SmolLM2 the suspect is `scatter_nd` + `equal` (bool) + `select` (bool cond) — the auxiliary mask/position-id ops. **Single root-cause investigation** would be cheap: capture both `mlmodelc/coremldata.bin` failures with `MLCustomConfig` ANE-only mode and `xcrun coremlcompiler` -v to get a per-op breakdown of which MIL op triggers the rejection. Could feed an upstream feature request to Apple or — more practically — a model-side rewrite hint for downstream LLM users.

3. **Inception V3 has 1 stranded `EinSum`.** Likely the multi-branch concat split point that creates the 2-CoremlOp shape. Adding it should give a single mega-CoremlOp.

4. **SmolLM2 has 2 stranded `Cast(I64)` ops.** Both around RoPE table indexing. Cast translator currently rejects I64 input dtype; lifting that restriction (with the same I64→I32 boundary cast we use on Gather) would close the last gap and give a truly whole-graph mega-CoremlOp.

5. **Parakeet/Nemotron joint stranded EinSums (3 each).** Final-head shape; probable axes-signature gap in `general_matmul`'s pattern matcher.

6. **Steady-state perf for ViT / YOLOv8 / RealESRGAN / SmolLM2 not yet measured.** Probes captured first-run latency only because the existing coverage tests don't call `state.run` in a loop. A small follow-up to add `*_perf_*` test functions would let those rows in the headline table get filled in. (No ANE engagement check yet either.)

## 4. ANE engagement summary

* **No ANE expected even though fused:**

  * RVM — 1 CoremlOp covers everything but Apple's scheduler refuses ANE on the recurrent loop and falls back to CPU+GPU.

## 5. Open questions / suggested follow-ups

1. **DFN3 6-CoremlOps regression** — was 3, now 6, around the same Scan boundaries. Net perf still acceptable (RTF 0.07) but worth a quick `git log` on the fusion/union-find changes to understand what split happened.
2. **Inception V3 single stranded EinSum** — likely the multi-branch concat split point that creates the 2-CoremlOp shape. Adding it should give a single mega-CoremlOp.
3. **SmolLM2 stranded I64 Casts** — `Cast input dtype I64 not supported`. Two instances, both around RoPE table indexing. Adding I64 to the Cast translator's supported-input dtypes would close the last gap and give a truly whole-graph mega-CoremlOp.
4. **Parakeet/Nemotron joint stranded EinSums** — 3 instances, final-head shape. Probable axes-signature gap; pattern-match should resolve.
5. **SAM 2 + SmolLM2 ANE compile failures** — different models, same `MILCompilerForANE FAILED` symptom. Worth a single root-cause investigation. Hypothesis (per phase-4-sam2-closure.md): rank-5/6 reshape patterns that ANE doesn't recognise.
6. **Parakeet/Nemotron decoder Scan loops** — would benefit from a "scan-body recursive transform" if we want to fold the per-timestep matmuls into ANE-eligible MLPackages.
7. **End-to-end perf for ViT, YOLOv8, RealESRGAN, SmolLM2** — first-run timings are present but no steady-state mean / RTF / sustained loops yet. If we want these in the PR table, we need either to add `*_perf_*` test functions or run a one-shot loop manually.
